# SEO & Social Metadata Overhaul

## Problem Frame

The Innovise site ships with minimal search and social metadata. The practical gaps:

- **No Open Graph or Twitter Card tags.** Links shared to LinkedIn, Slack, iMessage, or X render as bare URLs with no title, description, or preview image. For a firm whose entire positioning is "senior engineers, direct access, no handoffs," unformatted share previews actively undercut the brand.
- **No canonical URL.** Leaves the site exposed to duplicate-indexing of trailing-slash, `www`, and UTM-tagged variants.
- **Hand-maintained `public/sitemap.xml`** with a hardcoded `lastmod` of `2026-04-19`. The value is already stale and has no mechanism to update on deploy.
- **No structured data.** Organization, logo, and social profile associations are invisible to Google's knowledge graph.
- **Latent bug in `src/layouts/Layout.astro`**: favicon and webmanifest `href`s are relative (`href="images/logo/..."`) instead of root-absolute (`/images/logo/...`). These resolve correctly on `/` but break under any non-root path.

The remaining items (hero-image LCP optimization, font self-hosting) are real but carry more cost and are explicitly deferred.

---

## Requirements

**Social sharing metadata**
- R1. Every page renders full Open Graph tags: `og:title`, `og:description`, `og:type`, `og:url`, `og:image`, `og:image:alt`, `og:image:width`, `og:image:height`, `og:site_name`, `og:locale`.
- R2. Every page renders Twitter Card tags: `twitter:card` (`summary_large_image`), `twitter:title`, `twitter:description`, `twitter:image`, `twitter:image:alt`. `twitter:site` is intentionally omitted until a handle exists.
- R3. `og:title`/`og:description` default to the page's `<title>` and `<meta name="description">` but can be overridden per page via Layout props.
- R4. A production-ready share image exists at `/public/images/og-image.png` at 1200×630, referenced by both `og:image` and `twitter:image`. Image asset is supplied by the client, not generated by the implementation.

**Canonical and crawler metadata**
- R5. Every page renders `<link rel="canonical" href="...">` using the page's absolute URL derived from `Astro.site` and `Astro.url.pathname`.
- R6. `<meta name="theme-color">` is set using the Innovise brand color so mobile browser chrome matches the site.

**Structured data**
- R7. A JSON-LD `ProfessionalService` block is rendered in `<head>` on every page, including: `name`, `url`, `logo`, `description`, `address` (Bellevue, WA, US as `PostalAddress`), `sameAs: ["https://www.linkedin.com/company/innovise-technology"]`, and `founder` as a `Person` named Loren Anderson.
- R8. A JSON-LD `WebSite` block is rendered in `<head>` with `name`, `url`, and `publisher` referencing the same organization.
- R9. JSON-LD validates against Google's Rich Results Test with zero errors.

**Sitemap & robots**
- R10. `public/sitemap.xml` is deleted and replaced by the `@astrojs/sitemap` integration, so `lastmod` is generated at build time from source file mtimes.
- R11. `public/robots.txt` references the generated sitemap URL. No `Disallow` directives are added — everything remains indexable.

**Bug fixes rolled in**
- R12. All `href` values in `Layout.astro` under `<head>` (favicons, apple-touch-icon, webmanifest) use root-absolute paths (`/images/...`) instead of relative paths.

**Layout API**
- R13. `Layout.astro` accepts optional `title`, `description`, `image` (path relative to site root), and `noindex` props. When `noindex` is true, a `<meta name="robots" content="noindex">` is emitted and the page is excluded from the sitemap.

## Success Criteria

**Human outcomes**
- A link to `https://goinnovise.com` pasted into LinkedIn, Slack, iMessage, and X renders a formatted card with the brand's designed share image, a title, and a description.
- Google's Rich Results Test and the LinkedIn Post Inspector both report zero errors for the homepage.
- Search Console's "Enhancements" surface recognizes the Organization/ProfessionalService structured data.
- The sitemap's `lastmod` updates automatically on each deploy without anyone editing XML.

**Handoff quality**
- A downstream planner/implementer can execute without re-asking product decisions. All product choices (schema type, address, LinkedIn URL, OG image sourcing, whether to include Twitter card without a handle, whether to add `noindex` surfaces) are resolved in this doc.

## Scope Boundaries

- **Hero/LCP image optimization** — `hero-bg.png` is likely the LCP element and converting to WebP/AVIF and preloading it would help Core Web Vitals, but this is a real implementation lift and should be its own brainstorm.
- **Self-hosting Google Fonts** — measurable CWV win, but adds carrying cost. Out of scope.
- **Multiple indexable pages (blog, case studies, service pages)** — a product decision, not a meta-tag decision. Not addressed here.
- **Google Search Console / Bing Webmaster verification** — recommended to do via DNS, not HTML meta tags. No code change required, so out of scope for this workstream.
- **`security.txt` / `humans.txt`** — not SEO. Omit unless separately requested.
- **Keywords meta tag** — ignored by Google since 2009. Will not be added.
- **`twitter:site` handle** — deliberately omitted; add later if an X presence is created.
- **Breadcrumb / FAQ / Article schema** — inapplicable to a single-page marketing site. Would become relevant only if content pages are added.
- **Anchor-fragment URLs in the sitemap** (`#services`, `#team`) — Google ignores fragments for indexing. Not added.
- **Cookie banner / consent mode** — GA4 is already loaded unconditionally; that's a privacy/compliance decision for another brainstorm, not SEO.

---

## Key Decisions

- **Schema type: `ProfessionalService`** (not `Organization` or `LocalBusiness`). `ProfessionalService` is the closest fit for a consulting firm, and unlike `LocalBusiness` it does not require the address to imply a walk-in location.
- **Address included** as `Bellevue, WA, US`. Richer entity signal for Google; acceptable for a remote-first firm since `ProfessionalService` does not imply a physical storefront.
- **Social presence: LinkedIn only.** `sameAs` array contains exactly one URL. No GitHub or X entry because none currently exist as public org profiles.
- **Twitter Card included without `twitter:site`.** The card still renders correctly when others share the URL on X; omitting the handle is a one-line addition later.
- **Replace static sitemap rather than script its generation.** `@astrojs/sitemap` is the idiomatic path and removes a drift surface; rolling a custom generator is unwarranted.
- **Canonical uses `Astro.site` + `Astro.url.pathname`.** Avoids any risk of divergence between what's deployed and what's declared canonical.

---

## Dependencies / Assumptions

- **Client supplies `og-image.png`** (1200×630) before this ships to production. Until the asset exists, the OG tags can be wired up pointing at a path that doesn't yet resolve — but the work is not "done" until the image is in place.
- `Astro.site` in `astro.config.mjs` remains `https://goinnovise.com`. All canonical and OG URLs derive from it.
- The site stays single-page for the foreseeable future. If additional indexable pages are added, the Layout API in R13 already supports per-page overrides, so no rework is needed.
- No server-side rendering is introduced by this work. Everything remains static, compatible with the current Astro build.
