# Astro Resume System (Port from Jekyll Template)

## Problem Frame

Innovise currently maintains a separate Jekyll project (`Innovise/Resumes/loren/`) to publish a resume for each team member. The Jekyll template is data-driven (`_data/data.yml` with optional sections gated by Liquid `{% if %}` guards), but it is a dead-end stack relative to the rest of the company's web presence:

- It produces a separate site that visually does not match goinnovise.com — different fonts, different color tokens, Font Awesome iconography, two-column coral-sidebar layout.
- Adding a new person means creating a new Jekyll repo or directory, not adding a single file.
- The template's print CSS is thin (~38 lines) and predates the company's positioning work.
- Resume URLs live on a separate domain/path, so they don't share branding, structured data, analytics, or domain authority with the rest of the site.

The opportunity is to retire the Jekyll project entirely, serve resumes from the existing Astro site at `goinnovise.com/resumes/{slug}/`, and replace the per-person setup cost with "drop a YAML file in." The Astro site already has most of the seams in place: `Layout.astro` accepts a `noindex` prop and emits the correct robots meta; `astro.config.mjs` already anticipates a sitemap exclusion filter; brand tokens and image conventions are established; `loren-anderson.{webp,jpg}` already exists at `public/images/team/`.

A second deliverable: a Cursor skill that converts a third-party resume (pasted text or Markdown) into the Innovise YAML format with voice/tone aligned to the company's marketing-copy principles. This compounds the value of the system — every new hire's existing resume becomes a starting point rather than a blank file.

## Requirements

**Resume pages**
- R1. Resumes render at `/resumes/{slug}/` (e.g., `/resumes/loren/`). One page per person, generated by Astro at build time from a content collection.
- R2. Layout retains the two-column structure from the Jekyll template (sidebar with profile/contact/skills/etc., main column with career profile and experiences) but is restyled in the Innovise design system: `--font-heading`/`--font-body` tokens, brand color palette, sentence-case headings, no Font Awesome dependency, current `picture`-element image pattern.
- R3. Each resume page uses a trimmed page chrome — a header with the Innovise logo linking back to `/` (no menu) and a one-line footer with copyright and LinkedIn. Both the trimmed header and the footer are hidden in print.
- R4. The resume page emits `<title>`, `<meta name="description">`, canonical URL, Open Graph, and Twitter Card metadata derived from the resume's name, tagline, and career summary, consistent with the rest of the site's metadata pattern. (Page is `noindex`, so OG/Twitter primarily serve direct-link share previews, not search.)

**Data model & authoring**
- R5. Resumes are an Astro Content Collection located at `src/content/resumes/`, authored as YAML files (`{slug}.yaml`).
- R6. The collection is validated by a Zod schema. Required fields produce a build-time error if missing. Optional sections (e.g., education, projects, publications, languages, interests, phone) are explicitly typed as optional and are omitted from the rendered page when absent.
- R7. Schema covers, at minimum, the fields currently in the Jekyll `_data/data.yml`: profile (name, tagline, avatar path, optional location), contact (email required; phone, website, LinkedIn, GitHub, GitLab, Bitbucket, Twitter, Stack Overflow, Codewars, Goodreads — all optional), career profile (title + summary), experiences (role, time, company, location, summary paragraph, optional accomplishments list), education (optional, with optional `details`), skills (title + value pairs), and optional projects/publications/languages/interests sections.
- R8. Adding a new resume requires only: (a) one new YAML file, (b) one new headshot image, (c) no edits to existing components, layouts, or routes.

**Discoverability and indexing**
- R9. Every resume page renders `<meta name="robots" content="noindex, nofollow">`.
- R10. Resume URLs are excluded from `sitemap-index.xml` / `sitemap-0.xml` via the `filter` option on the `@astrojs/sitemap` integration.
- R11. There is no `/resumes/` index page. There are no links to resumes from `Navbar`, `Footer`, or any other site page. Resume URLs are direct-share artifacts only.
- R12. `public/robots.txt` does not need to change — the per-page `noindex` meta is sufficient and consistent with the rest of the site's posture. (`Disallow` would be the wrong tool here: it blocks crawl, but a disallowed URL can still be indexed if linked elsewhere; allow-crawl-with-noindex is the cleaner posture.)

**Print & PDF**
- R13. Each resume page includes an on-screen "Print / Save as PDF" button styled to match the design system. The button is hidden in print output.
- R14. Print CSS supports both US Letter and A4 page sizes, with the default `@page { size: letter; margin: 0.5in; }` set in CSS and the user free to override via the browser's print dialog.
- R15. Print output hides the trimmed page header, the trimmed page footer, the on-screen Print button, and any animation/parallax hooks. The sidebar's brand-color background prints as a thinner monochrome accent strip (or grayscale) to save ink and avoid the heavy coral block dominating a printed document. Section title color and any link underlines are preserved using `-webkit-print-color-adjust: exact`.
- R16. Per-experience blocks use `page-break-inside: avoid` so a single role does not split awkwardly across two printed pages. Section headings use `page-break-after: avoid` so a heading never lands at the bottom of a page.

**Resume-author Cursor skill**
- R17. A Cursor skill (`.cursor/skills/resume-author/`) accepts a third-party resume as plain pasted text or a Markdown file as its input. Other formats (PDF, .docx, LinkedIn JSON export) are explicitly out of scope for v1.
- R18. The skill produces a complete `src/content/resumes/{slug}.yaml` file conforming to the Zod schema, plus instructions to the user about where to drop the candidate's headshot image (e.g., `public/images/resumes/{slug}.jpg`).
- R19. The skill applies the **transferable** principles from `.cursor/rules/marketing-copy.mdc` to the candidate's prose: anti-jargon ("synergy", "leverage", "best-in-class", "end-to-end solutions"), sentence case for headings, specificity over adjectives, directness, no inflated symbolism. It does **not** inject firm-positioning claims that belong to Innovise (e.g., "senior engineers, start to finish", first-call process language, location boilerplate).
- R20. The skill never invents facts. Dates, titles, companies, technologies, and accomplishments must come from the source document. When a fact is unclear, missing, or ambiguous, the skill asks the user a clarifying question rather than guessing.
- R21. The skill operates interactively, asking one focused question at a time when source data is incomplete or ambiguous (missing dates, unidentifiable role title, unclear technologies, weak career-summary content, no explicit headshot path).
- R22. The skill cites which transferable marketing-copy principles it applied during the rewrite, so the user can review specific edits rather than diffing the whole file blind.

## Success Criteria

- A senior engineer being onboarded next quarter can have a published resume by adding a single YAML file and a single image, with no template or layout edits.
- The Jekyll resume project (`Innovise/Resumes/loren/`) can be archived or deleted without losing any production capability.
- Loren's resume at `goinnovise.com/resumes/loren/` is content-equivalent to the current Jekyll resume, visually consistent with the rest of goinnovise.com, and prints cleanly to a one-or-two-page PDF in both Letter and A4 sizes.
- The Cursor skill, run on a representative sample third-party resume (e.g., a generic mid-career engineer's resume), produces a YAML file that compiles, renders correctly, preserves all source-of-truth facts, and reads in a voice consistent with the rest of the site.
- A Search Console crawl 30 days post-launch confirms zero resume URLs indexed and zero "Submitted URL marked noindex" warnings (by virtue of the sitemap exclusion).

## Scope Boundaries

- No `/resumes/` index or directory page is built. Resume URLs are direct-share only.
- No links to resumes from the main navigation, footer, or `Team` component. Surfacing resumes from elsewhere on the site is a separate decision later.
- No build-time PDF generation (Puppeteer / headless Chrome). Browser print-to-PDF is the PDF story for v1.
- No automated migration of existing Jekyll resume HTML. Loren's resume is converted via the new system end-to-end (manually or via the new skill on his existing `data.yml`); the Jekyll project is then retired.
- No PDF / .docx / LinkedIn-export ingestion in v1 of the Cursor skill. Plain text and Markdown only.
- No skill-driven rewriting of facts. The skill is voice-aligned, not embellishing.
- No team directory features (filtering, sorting, search). One file in, one URL out.
- The marketing copy on the Innovise homepage's `Team` block remains independent of resume data — these are different artifacts with different audiences, allowed to drift.

## Key Decisions

- **Hybrid visual approach: two-column structure, Innovise design system styling.** The two-column layout is a strong resume convention worth keeping. The Jekyll-era visual treatment (Font Awesome icons, Roboto font, raw coral sidebar) is replaced with the site's design tokens: IBM Plex Sans for headings, Source Sans 3 for body, the existing `--color-brand-*` palette, current `picture`-element image conventions, no icon-font dependency.
- **Astro Content Collection with Zod schema, YAML files.** The native Astro authoring model gives compile-time validation on every build. YAML keeps the editing experience close to the existing `data.yml` so authors moving from Jekyll feel no friction. Markdown frontmatter was rejected because resume data is overwhelmingly structured, not prose.
- **`noindex` per page, excluded from sitemap.** Google's documented advice is not to list noindex URLs in a sitemap. Including them with noindex would produce persistent Search Console warnings without changing what gets indexed. The existing comment in `astro.config.mjs` already anticipates this exact filter pattern.
- **Direct-share only, no `/resumes/` index.** Resumes are credentials shared with specific prospects, not a browseable directory. An index page would invert the noindex framing and reintroduce a team-directory question we've otherwise scoped out.
- **Trimmed header + minimal footer, both hidden in print.** Full Navbar (`position: fixed`) fights printability and out-competes the document for attention. No chrome at all loses brand framing for direct-link visitors. The trimmed approach is the balance.
- **Phone is optional in schema, included on Loren's resume.** User accepted the cell-phone exposure trade-off for parity with the existing Jekyll resume. Future hires can omit by leaving the field blank.
- **Letter + A4 print rules.** A small additional cost up front, future-proofs for non-US hires. Browser will rescale anyway if the user picks a non-default size at print time.
- **Skill is voice-aligned, never inventive, interactive when ambiguous.** This is the safest default for an artifact going on the public site under a real person's name. The skill's failure mode should be "asks too many questions," not "fabricates a credential."

## Dependencies / Assumptions

- Astro Content Collections feature (stable in Astro 6.x, already on the site's dependency tree) is the foundation. No new runtime dependency required for the resume system itself.
- The `@astrojs/sitemap` integration's `filter` option is the supported mechanism for excluding the resume URLs. This is already in `astro.config.mjs` as a comment; the implementation is a one-liner.
- The Cursor skill assumes the user has an active Cursor session and can invoke skills. No external services or API keys.
- Schema validation surfaces errors at build time, which is acceptable because the build pipeline is the deploy gate.
- "Voice-aligned per marketing-copy.mdc" assumes the marketing-copy rules continue to be the single source of truth for Innovise voice. If those rules change, the skill's outputs will reflect them on next run.
